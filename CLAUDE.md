## Project Overview

`air-monitor` is a JavaScript (ES module) **library** providing a compact
in-memory data model — the `Monitor` class — for working with hourly air quality
monitoring data, with a focus on PM2.5. It loads CSVs (from the USFS AirFire S3
archive or custom URLs), then transforms, summarizes, filters, and exports them
as arrays, plain objects, GeoJSON, or new `Monitor` instances. Heavy scientific
algorithms (NowCast, daily/diurnal statistics) are delegated to the companion
`air-monitor-algorithms` package.

- **Consumers:** scientific and data-visualization developers, including Svelte
  and Vue applications.
- **Primary goals:** correctness and reliability of the data model first, then
  maintainability and clarity. Not all contributors are professional software
  engineers, so readable code and clear documentation matter.
- **Distribution:** published to npm as `air-monitor` (GPL-3.0-or-later). The
  version number lives in `package.json`; `NEWS.md` is the human-readable
  changelog.
- **Maturity:** stable, actively maintained. Recent work has been focused
  bug-fixing and documentation, not new features.

See `.claude/CLAUDE_ARCHITECTURE.md` for the module map, call graph, data
conventions, and public API contract.

## Language and Style

- Modern JavaScript (ES2020+), native ES modules (`"type": "module"`). No
  TypeScript.
- Prefer `async`/`await` over promise chains.
- Every public method on `Monitor` and every exported function carries **JSDoc**
  with `@param`, `@returns`, units where relevant, and `@throws` where it can
  throw. Keep the JSDoc return type honest — e.g. the `load*()` methods mutate
  in place and resolve to `void`, not to a `Monitor`.
- Data manipulation is done with **Arquero** table operations. Datetime handling
  is done with **Luxon** `DateTime` objects (UTC on the wire; explicit IANA
  timezone for local-time operations).
- Validate inputs and fail loudly with a clear, contextual message rather than
  failing silently — see `validateDeviceID`, `validateDataTable`, `parseDatetime`
  in `src/utils/helpers.js` for the house style.
- Public `Monitor` methods are thin wrappers that delegate to `internal_*`
  functions in `src/utils/`. Keep that split: the class is the API surface; the
  logic lives in the utility modules.

## Project-Specific Constraints

### Data Conventions (load-bearing — every function relies on these)

- A monitor is **two aligned Arquero tables**: `meta` (one row per
  `deviceDeploymentID`) and `data` (rows = `datetime`, columns =
  `deviceDeploymentID`s). The shared `deviceDeploymentID` is the join key linking
  a `meta` row to its `data` column.
- `data.datetime` holds **Luxon `DateTime` objects in UTC**, strictly increasing,
  spaced **exactly 1 hour apart, with no gaps**. This is enforced by
  `validateDataTable()`.
- All non-`datetime` values are finite numbers or `null`. `'NA'`, `NaN`, and
  non-finite values are normalized to `null` during parsing.
- Negative PM2.5 measurements are clamped to `0` (a domain rule in `parseData`).
- `deviceDeploymentID` is unique within `meta`. When `collapse()` creates a new
  series, its `deviceDeploymentID` MUST equal the single data column name, or
  every accessor breaks (see the regression note in `internal_collapse`).

### Public API and Compatibility

The public contract is the `Monitor` class and its method signatures / return
shapes (see `src/Monitor.js` and the Public API Contract section of the
architecture doc). Changing a method name, parameter order, return shape, or unit
is a **breaking change** that requires a version bump in `package.json` and a
`NEWS.md` entry. Transformation methods return a **new** `Monitor`; `load*()`
methods mutate `this` in place.

### Key Dependencies (do not add or replace without discussion)

- `arquero` (pinned `7.2.1`) — the table engine; the entire data model is built
  on it.
- `luxon` (pinned `3.6.1`) — all datetime handling; the UTC/timezone contract
  depends on it.
- `air-monitor-algorithms` — NowCast and daily/diurnal statistics. Keep the
  scientific algorithms there, not here.

Avoid adding dependencies unless they provide significant value.

### Domain-Specific Rules

- Do not hand-edit anything in `dist/` — it is generated by `npm run build`.
- Do not hand-edit `docs/` — it is generated by `npm run docs`.
- Preserve the UTC / hourly / gap-free datetime invariant; many functions assume
  it and `validateDataTable()` guards it.

## Build, Test, and Document

- `npm run build` — bundle `src/` into `dist/` (ESM + UMD) via Rollup. `dist/` is
  committed.
- `npm test` — run the UVU test suite in `tests/`.
- `npm run docs` — generate JSDoc HTML into `docs/`.
- `npm run copy-to-vendor` — refresh the vendored algorithms bundle used by the
  browser examples.

New or changed behavior should come with tests (the suite already has regression
tests for previously fixed bugs; follow that pattern). Run `npm test` before any
version bump or publish.

## Review Expectations

When reviewing code:

1. Identify correctness issues.
2. Identify operational risks and backward-compatibility concerns.
3. Identify documentation gaps.
4. Suggest low-risk improvements.

Prioritize and communicate recommendations as described in
`.claude/CLAUDE_STYLE_GUIDE.md` (Review Priorities and Communication Style).
Do not assume a rewrite is desired.
